# AI Agent 개발을 잘하려면

> 백엔드 개발자를 위한 Agent 개발 역량 로드맵

---

## 1. 기존 백엔드 역량이 그대로 자산이다

Agent 개발은 완전히 새로운 게 아니다. 오히려 **기존 백엔드 역량이 핵심**이다.

| 기존 역량 | Agent 개발에서의 역할 |
|---|---|
| API 설계 능력 | Tool 설계 능력으로 직결 |
| 시스템 설계 | Multi-Agent 아키텍처 설계 |
| 비동기/동시성 | 여러 Agent 병렬 실행 관리 |
| 에러 처리 | LLM 응답 실패, Tool 실패 핸들링 |
| 보안 감각 | Prompt Injection 방어, Tool 권한 제어 |

---

## 2. 새롭게 익혀야 할 것

기존 역량 위에 **이것들만 추가로** 익히면 된다.

### 2-1. 프롬프트 엔지니어링

LLM에게 역할, 제약, 출력 형식을 명확하게 지시하는 능력.
코드로 비유하면 **함수 시그니처와 주석을 잘 쓰는 것**과 같다.

```
나쁜 프롬프트:
  "주식 분석해줘"

좋은 프롬프트:
  "너는 퀀트 애널리스트다. 입력받은 종목의 최근 30일
  가격 데이터를 기반으로 추세, 변동성, 지지선을 분석해라.
  반드시 JSON 형식으로 응답해라."
```

**잘 쓴 프롬프트의 조건:**
- 역할(Persona) 명확히 부여
- 입력/출력 형식 명시
- 해야 할 것과 하지 말아야 할 것 구분
- 예시(Few-shot) 포함 시 품질 향상

---

### 2-2. Tool 설계

LLM이 Tool을 올바르게 쓰려면 **Tool의 이름, 설명, 파라미터가 명확**해야 한다.
REST API 문서 잘 쓰는 것과 완전히 같은 개념.

```typescript
// 나쁜 Tool 설계
{
  name: "get_data",
  description: "데이터를 가져온다",
  parameters: { id: { type: "string" } }
}

// 좋은 Tool 설계
{
  name: "get_stock_price",
  description: "특정 종목의 현재 주가와 전일 대비 등락률을 조회한다. 실시간 데이터 기준.",
  parameters: {
    ticker: {
      type: "string",
      description: "종목 코드. 국내는 6자리 숫자(e.g. 005930), 미국은 알파벳(e.g. AAPL)"
    }
  }
}
```

**LLM은 Tool의 description을 읽고 언제 쓸지 판단한다. Tool 설명이 곧 품질이다.**

---

### 2-3. 평가 (Evals)

LLM 기반 시스템은 단위 테스트만으로는 부족하다.
"이 Agent가 올바른 판단을 하는가"를 측정하는 **평가 파이프라인** 구축이 중요하다.

| 테스트 종류 | 기존 백엔드 | Agent |
|---|---|---|
| 정확성 검증 | 단위 테스트 (assert) | Eval (정답 셋과 LLM 출력 비교) |
| 회귀 테스트 | CI/CD 자동화 | 프롬프트 변경 시 Eval 재실행 |
| 성능 측정 | 응답 시간, throughput | 정확도, 비용, 토큰 사용량 |

아직 많은 팀이 Eval을 제대로 구축하지 못하고 있어서, **여기서 차별화**가 생긴다.

---

### 2-4. 비용/지연 최적화

LLM API는 토큰당 과금이라 잘못 설계하면 비용이 폭증한다.

- **불필요한 LLM 호출 줄이기** → 단순 조회는 LLM 없이 직접 처리
- **캐싱 전략** → 동일한 입력에 대한 LLM 응답 캐시 (Redis 활용)
- **모델 분기** → 간단한 작업은 작은 모델, 복잡한 추론만 큰 모델 사용
- **컨텍스트 관리** → 불필요한 이전 대화 히스토리 자르기

## 3. 학습 순서 추천

```
1단계  LLM API 직접 사용해보기
       → OpenAI / Claude API로 간단한 챗봇 만들어보기
       → Tool Calling (Function Calling) 프레임워크 없이 직접 구현
       → 원리를 이해하는 게 목적

2단계  Agent 프레임워크 사용
       → LangChain, LangGraph, 또는 Claude Agent SDK
       → 1단계에서 직접 만든 Tool을 프레임워크에 연결
       → 생산성과 구조화된 패턴 학습

3단계  실제 프로젝트에 적용
       → 지금 작업 중인 서비스에 작은 자동화 기능 하나 추가
       → 완벽하게 만들려 하지 말고, 동작하는 것부터 시작

4단계  평가 체계 구축
       → 잘 동작하는지 측정하는 파이프라인 만들기
       → 프롬프트 버전 관리 (코드처럼 git으로 관리)
       → 지표 기반으로 개선 반복
```

---

## 4. 주요 프레임워크

| 프레임워크 | 언어 | 특징 | 추천 용도 |
|---|---|---|---|
| LangChain | Python / JS | 가장 범용적, 생태계 풍부 | 빠른 프로토타이핑 |
| LangGraph | Python / JS | 그래프 기반 워크플로우 | 복잡한 Multi-Agent |
| CrewAI | Python | Multi-Agent 특화, 직관적 | 역할 분담 자동화 |
| Claude Agent SDK | Python | Anthropic 공식 | Claude 모델 중심 서비스 |
| Vercel AI SDK | TypeScript | Next.js 친화적 | 웹 서비스 통합 |

> 처음엔 프레임워크 없이 LLM API 직접 호출로 시작하는 걸 권장한다.
> 원리를 모르면 프레임워크가 블랙박스가 된다.

## 5. 앞으로 개발자 역할 변화

| 지금 | 앞으로 |
|---|---|
| 모든 로직을 코드로 직접 구현 | 반복/판단 로직은 Agent에 위임 |
| 기능 개발 중심 | Tool 설계 + Agent 오케스트레이션 중심 |
| 단위 테스트 | 단위 테스트 + LLM 평가(Evals) |
| 혼자 코딩 | AI와 페어 프로그래밍 (Claude Code 등) |
| API 엔드포인트 설계 | Tool 인터페이스 설계 |

---

## 6. 핵심 마인드셋

### "LLM을 잘 쓰는 개발자"가 아니라
### "LLM이 잘 쓸 수 있는 시스템을 설계하는 개발자"가 목표다.

LLM 자체의 성능은 모델 회사가 올려준다.
개발자의 역할은 **LLM이 올바른 판단을 내릴 수 있도록 환경을 설계하는 것**이다.

- 명확한 Tool 정의
- 적절한 컨텍스트 제공
- 실패 케이스 처리
- 지속적인 평가와 개선

이것은 결국 **좋은 시스템 설계**와 같다. 백엔드 개발자가 가장 잘할 수 있는 영역이다.

---

## 7. 지금 당장 시작하는 법

> 가장 빠른 학습법은 **지금 작업 중인 프로젝트에서 반복적인 작업 하나를 골라 Agent로 자동화**해보는 것이다.

예시:
- 매일 특정 종목 시황 분석 후 슬랙 발송 자동화
- 이상 거래 감지 후 알림 Agent
- 자연어로 종목 검색 및 분석 인터페이스

작게 시작해서, 동작하면 점점 확장하라.
